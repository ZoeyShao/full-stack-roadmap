认证和授权不要混

概念问题例子
Authentication你是谁？登录成功，系统知道 user id。
Authorization你能做什么？只有项目 owner 能删除项目。
Audit谁在什么时候做了什么？记录管理员修改权限的事件。

Session vs JWT

方式适合风险关键实践
Session cookie传统 Web 应用，同域前后端。CSRF、会话固定、cookie 配置错误。HttpOnly、Secure、SameSite、轮换 session id。
JWT access tokenAPI、移动端、短期访问 token。泄漏后难撤销、把敏感信息放入 payload。短过期、验证签名和 claims、最小 claims。
Refresh token延长登录状态。长期 token 泄漏。安全存储、轮换、撤销列表、设备绑定。

JWT 签名只能证明 token 没被篡改，不代表内容保密。不要把密码、身份证、密钥放进 JWT payload。

OAuth 授权码流程

user
  -> client app: 点击第三方登录
  -> authorization server: 授权请求
  -> user: 登录并同意
  -> client app: 收到 authorization code
  -> authorization server: 用 code 换 token
  -> resource server: 用 access token 访问资源

OAuth 的核心是授权，不是“密码交给第三方”。现代 Web 应用还应理解 PKCE、redirect URI 校验和 state 防 CSRF。

OWASP 项目自查

风险自问
Broken Access Control后端是否每次都检查资源所有权，而不是只靠前端隐藏按钮？
Cryptographic Failures密码是否用专用哈希，传输是否 HTTPS，密钥是否进了仓库？
InjectionSQL 是否参数化，用户输入是否被拼进命令或查询？
Security Misconfiguration默认密码、公开 bucket、详细错误堆栈是否暴露？
Vulnerable Components依赖是否有漏洞扫描和更新策略？
