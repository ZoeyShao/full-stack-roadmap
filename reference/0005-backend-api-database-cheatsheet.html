HTTP API 请求处理链

client request
  -> load balancer / gateway
  -> route match
  -> authentication
  -> validation
  -> business logic
  -> database
  -> response mapping
  -> logs / metrics

后端不是“写几个接口”，而是把输入、权限、业务规则、数据一致性和可观测性放在一个稳定边界里。

API 设计检查清单

检查项问题
资源URL 表示名词资源，还是把动词塞进路径了？
验证缺字段、格式错、越权字段如何返回？
错误错误响应是否有稳定结构和可读 message？
分页列表是否限制 limit，是否支持 cursor 或 page？
幂等重复请求是否会导致重复创建或重复扣费？
版本未来 breaking change 如何处理？

关系型数据库建模

users
  id primary key
  email unique not null
  created_at timestamp

projects
  id primary key
  owner_id foreign key -> users.id
  title not null
  status check ("draft", "active", "archived")

tasks
  id primary key
  project_id foreign key -> projects.id
  title not null
  completed_at nullable timestamp

先找实体，再找关系，再找约束。数据库约束不是麻烦，它是系统底线。

SQL 查询模板

SELECT p.id, p.title, count(t.id) AS task_count
FROM projects p
LEFT JOIN tasks t ON t.project_id = p.id
WHERE p.owner_id = $1
GROUP BY p.id, p.title
ORDER BY p.created_at DESC
LIMIT 20;
语句作用
WHERE过滤行，尽早减少数据。
JOIN按关系合并表。
GROUP BY按维度聚合。
ORDER BY排序，常与索引和 limit 相关。
LIMIT限制返回数量，避免列表无界。
