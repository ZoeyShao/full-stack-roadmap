系统设计回答框架

1. clarify requirements
2. define APIs and core flows
3. model data
4. estimate scale
5. draw high-level architecture
6. choose storage, cache, queue, compute
7. handle failure, security, observability
8. call out tradeoffs and next iterations

常见取舍

选择得到付出
缓存低延迟、少打数据库。失效策略、数据过期、一致性复杂度。
队列削峰、异步、解耦。最终一致性、重试、重复消费。
读写分离读扩展能力。复制延迟、读到旧数据。
Serverless少运维、按需扩展。冷启动、运行时限制、可观测性细节。
微服务团队独立交付、边界清晰。网络故障、分布式追踪、部署复杂度。

项目展示清单

位置必须有加分项
README问题、功能、技术栈、运行方式、截图。架构图、测试说明、部署地址。
Demo2 分钟核心流程录屏。先讲用户问题，再讲技术取舍。
代码清晰目录、环境变量模板、迁移脚本。CI badge、测试覆盖、监控截图。
作品集项目卡片、链接、你的职责。技术深挖页、系统设计说明。

简历 bullet 模板

Built [feature/system] using [technology],
improving [metric/outcome] by [number],
while handling [constraint/tradeoff].

中文表达可以是：使用 X 构建 Y，在 Z 约束下实现 A 结果。不要只列“熟悉 React、熟悉 Node”。

项目面试故事

问题你的回答要包含
为什么做这个项目？用户问题、业务场景、目标约束。
最难的技术点？失败尝试、最终方案、权衡。
如何保证质量？测试、类型、CI、监控、手动验收。
上线后怎么排障？日志、指标、告警、request id、回滚。
下次会怎么改？承认边界，给出优先级和可执行改进。
