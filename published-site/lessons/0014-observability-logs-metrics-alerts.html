0014 日志、指标和告警

Q3 Cloud + DevOps | 预计 35 分钟 | lecture trace

上线后的系统不会自己解释故障。你要提前设计它如何说话。

三种信号

信号回答例子
Logs这次请求发生了什么？request_id、user_id、route、status、error。
Metrics整体趋势如何？QPS、p95 latency、5xx rate。
Traces跨服务时间花在哪里？frontend -> API -> DB。

告警原则

  • 告警应该对应用户影响或明确行动。
  • 只对症状告警，不要只对原因猜测告警。
  • 每条告警都要有 runbook。
  • 错误率、延迟、饱和度通常比单条错误日志更适合告警。

练习

  1. 为 API 设计结构化日志字段。
  2. 设计 5xx rate、p95 latency、auth failure 三个指标。
  3. 写一条告警触发条件。
  4. 写故障时第一步排查动作。
交付物：一份最小观测方案。

自测

日志和指标有什么区别？

日志记录具体事件细节，指标记录可聚合的数值趋势。

为什么告警不能太多？

告警噪音会让人忽略真正重要的问题，降低响应质量。

SLO 的作用是什么？

把可靠性目标量化，让团队能围绕用户体验做取舍。
