0009 SQL 数据建模与查询

Q2 Backend Fundamentals | 预计 40 分钟 | lecture trace

数据库不是后端的附属品。它保存业务事实，约束错误数据，并决定很多性能边界。

建模顺序

  1. 找实体：User、Project、Task。
  2. 找唯一身份：每张表一个 primary key。
  3. 找关系：Task 属于 Project，Project 属于 User。
  4. 找约束：必填、唯一、状态枚举、外键。
  5. 找查询：先由页面和 API 反推需要哪些读取路径。

例子

CREATE TABLE projects (
  id uuid PRIMARY KEY,
  owner_id uuid NOT NULL REFERENCES users(id),
  title text NOT NULL,
  status text NOT NULL CHECK (status IN ('draft', 'active', 'archived')),
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX projects_owner_created_idx
ON projects(owner_id, created_at DESC);

查询直觉

工具什么时候用
JOIN跨表读取关联数据。
INDEX高频过滤、排序或关联字段。
TRANSACTION一组操作必须一起成功或失败。
EXPLAIN理解数据库如何执行查询。

练习

  1. 设计 users、projects、tasks 三张表。
  2. 写一个查询：返回某用户所有项目及任务数。
  3. 说明你会给哪些字段建索引，为什么。
  4. 设计“创建项目并创建默认任务”的 transaction。
交付物：schema、5 条 SQL、索引说明。

自测

主键和外键分别保护什么？

主键保护行的唯一身份；外键保护表之间引用的完整性。

索引是不是越多越好？

不是。索引加快某些读，也增加写入和存储成本。

事务解决什么问题？

保证一组相关数据库操作在失败时不会留下半完成状态。
