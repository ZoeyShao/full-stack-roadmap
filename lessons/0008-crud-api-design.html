0008 CRUD API 设计

Q2 Backend Fundamentals | 预计 35 分钟 | lecture trace

CRUD 是业务系统的骨架。好的 API 合约让前端、后端、测试和未来维护都少猜一点。

资源和方法

动作方法路径状态码
列表GET/projects200
详情GET/projects/:id200, 404
创建POST/projects201, 400, 409
更新PATCH/projects/:id200, 400, 404
删除DELETE/projects/:id204, 404

边界行为

  • 验证失败返回 400，并指出字段问题。
  • 资源不存在返回 404，不要返回空对象假装成功。
  • 列表必须分页，避免无界响应。
  • 删除和重试要考虑幂等。
  • 错误结构稳定，便于前端展示。

练习

  1. 为 projects 写出 6 个 endpoint。
  2. 为创建接口写成功响应和 3 个失败响应。
  3. 给列表接口加 limit 和 cursor。
  4. 说明 DELETE 重复调用应该怎样表现。
交付物：一份 CRUD API 合约。

自测

为什么 URL 里尽量用名词？

REST URL 表示资源，动作交给 HTTP method 表达。

POST 和 PATCH 的区别是什么？

POST 常用于创建或触发非幂等动作；PATCH 用于部分更新已有资源。

为什么列表要分页？

避免响应过大、查询过慢和前端一次渲染太多数据。
